Design — Section 1: Data Model & Architecture

How budgets are stored and computed. Does this look right?

New SQLite table: budgets

id TEXT PRIMARY KEY
category_id TEXT -- NULL = overall budget
amount_limit REAL -- e.g. 20000.00
currency TEXT -- e.g. "INR"
period TEXT -- "monthly" (only mode)
created_at INTEGER
updated_at INTEGER

One row per budget. category_id = NULL = the overall monthly cap. Each category gets its own row. Budget history is preserved — old rows are never deleted, new ones are inserted each month.

Computed query (spent vs limit)

SELECT b.id, b.category_id, b.amount_limit,
  COALESCE(SUM(t.amount), 0) AS amount_spent
FROM budgets b
LEFT JOIN transactions t
  ON (b.category_id IS NULL OR t.category_id = b.category_id)
  AND t.flow = 'OUT'
  AND t.created_at BETWEEN startOfMonth AND now
GROUP BY b.id

Runs live against the existing transactions table. No duplication. Uses same date-fns already in the project for month boundaries.

action

computed

computed

Settings additions

New fields added to existing settingsStore.ts:

budgetAlerts: {
  threshold75: boolean + number (default 75)
  threshold90: boolean + number (default 90)
  threshold100: boolean (always recommended on)
  weeklyDigest: boolean
}

Thresholds are numbers the user can edit (e.g. change 75 → 80). Each has a toggle. Persisted in existing AsyncStorage-backed settingsStore — no new store needed.
